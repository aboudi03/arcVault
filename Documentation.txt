How the pieces connect: 
A Make.com webhook receives incoming support requests. An Iterator splits the batch into one message at a time. Each message is sent to the LLM through an HTTP request, and the model returns structured JSON with classification and enrichment fields. A Parse JSON step turns that response into usable fields. The first router checks for low-confidence cases and sends them to human escalation. If confidence is acceptable, the second router maps the category to the correct queue. Finally, Airtable stores the structured record with the category, priority, confidence score, extracted fields, destination queue, escalation flag, and summary. I intentionally kept the routing and escalation logic deterministic outside the LLM. The model is used for classification and enrichment, while routing decisions remain rule-based inside the workflow. This ensures that business rules remain transparent, testable, and easy to modify without changing the prompt or relying on the model to make operational decisions.


Routing logic:
I kept the routing logic simple and deterministic after the LLM step so the final destination is easy to understand and control. Bug reports go to Engineering because they usually point to something broken in the product. Feature requests go to Product because they are requests for new functionality, not defects. Billing issues go to Billing because they involve invoices, charges, or pricing questions. Technical questions go to IT/Security because they are usually setup or configuration questions, such as SSO or authentication. Incident and outage cases go to Human Escalation because they can affect multiple users and need faster review than normal tickets.

Escalation logic:
I implemented escalation primarily for low-confidence and incident-style cases. Any request with a confidence score below 70 is routed to Human Escalation because the model is signaling uncertainty and a human should review it before action is taken. I also escalate Incident/Outage cases because they often imply multiple-user impact, urgency, and higher business risk. This logic is conservative by design: the cost of escalating one extra ambiguous request is lower than the cost of missing a real outage or mishandling a low-confidence classification.

What I would do differently at production scale (reliability, cost, latency):

At production scale, I would separate ingestion from processing using a queue-based architecture so bursts of incoming requests do not overwhelm the workflow. A queue system such as Redis with BullMQ would allow retries, failure handling, deduplication, and horizontal scaling of worker processes as traffic grows.
I would also store results in a production database instead of Airtable so the system becomes more reliable, easier to query, and better suited for downstream analytics.
To improve cost and latency, I would introduce model tiering: simpler requests would be handled by a smaller or cheaper model, while uncertain or complex cases would be retried with a stronger model.
Finally, I would improve prompt evaluation and calibration over time by collecting labeled examples of support tickets. This dataset could be used to validate classification accuracy and better calibrate confidence scores, particularly distinguishing between genuinely low-confidence classifications and high-risk cases such as outages.

What I would add in a Phase 2 if I had another week:
In a second phase, I would focus on reliability, validation, and more use cases which i have to test also, I would add monitoring, better error handling, and a lightweight human review flow for escalated records. I would also add alerting and grouping logic so that if many similar requests arrive within a short time window, the workflow can treat them as a potential incident rather than separate unrelated tickets. This would improve incident detection and help the team respond faster to broader service issues.
